    linux 出错 task xxxxx 634 blocked for more than 120 seconds

    2024年3月19日, 4分钟阅读

    • #故障排除和性能优化

    问题现象

    • 系统挂起，无法响应
    • SSH无响应，但可以ping通

    报错中通常还有关键信息“hung_task_timeout_secs”，linux kernel bug，相关描述

    By default Linux uses up to 40% of the available memory for file system caching. After this mark has been reached the file system flushes all outstanding data to disk causing all following IOs going synchronous. For flushing out this data to disk this there is a time limit of 120 seconds by default. In the case here the IO subsystem is not fast enough to flush the data withing 120 seconds. This especially happens on systems with a lot of memory. The problem is solved in later kernels

    一般情况下，linux会把可用内存的40%的空间作为文件系统的缓存。当缓存快满时，文件系统将缓存中的数据整体同步到磁盘中。但是系统对同步时间有最大120秒的限制。如果文件系统不能在时间限制之内完成数据同步，则会发生上述的错误。这通常发生在内存很大的系统上。系统内存大，则缓冲区大，同步数据所需要的时间就越长，超时的概率就越大

    原因分析

    缺乏操作系统资源或配置错误

    解决方案

    修改系统设置

    调整/proc/sys/kernel/hung_task_timeout_secs值以禁用此警告信息

    检查硬件资源

    确保内存和交换空间充足

    更新或修复Linux内核

    关注Linux kernel的bug修复

    使用特定工具或命令行操作

    查看系统日志以定位具体问题

    预防措施

    监控系统性能指标 使用监控工具定期检查系统负载和资源使用情况

    解决这个问题的方法可能包括：

    1. 检查并增加系统资源，如增加内存或扩大交换空间，以确保有足够的资源来启动新的进程或服务
    2. 调整内核参数hung_task_timeout_secs，通过设置为0可以禁用这个消息。这表明系统认为某个任务已经挂起太长时间，但同时也意味着系统无法自动处理这种情况
    3. 如果问题是由于内核bug引起的，可能需要等待内核更新或补丁来解决问题

    此外，当遇到此类问题时，还应该检查系统的其他日志文件和系统状态，以确定是否有其他相关的问题或错误。例如，如果服务器突然无反应，可以尝试重启服务器，并查看日志以获取更多信息9。

    总之，面对“INFO: task xxxxxx: 634 blocked for more than 120 seconds.”的错误，需要综合考虑系统资源、配置以及可能存在的内核问题，采取相应的解决措施。
